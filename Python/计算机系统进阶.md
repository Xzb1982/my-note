在冯诺依曼的计算机架构上，我们进一步细化CPU是怎么进行计算的？
	几乎所有冯诺依曼架构的CPU都进行五个工作阶段：
		取指令、分析指令、执行指令、访存取数、结果写回
		取指令IF Instruction Fetch
			是将一条指令从主存中取到指令寄存器的过程。
			程序计数器 PC 中的数值，用来指示当前指令在主存中的位置。
			当一条指令被取出后，PC 中的数值将根据指令字长度而自动递增：若为单字长指令，则(PC)+1->PC；若为双字长指令，则(PC)+2->PC，依此类推。
		指令译码（Instruction Decode，ID）
			[指令译码器](https://zhida.zhihu.com/search?content_id=185093927&content_type=Article&match_order=1&q=%E6%8C%87%E4%BB%A4%E8%AF%91%E7%A0%81%E5%99%A8&zhida_source=entity)按照预定的指令格式，对取回的指令进行拆分和解释，识别区分出不同的指令类别以及各种获取操作数的方法。
		执行指令（Execute，EX）
			 此阶段的任务是完成指令所规定的各种操作，具体实现指令的功能。
			 为此，CPU 的不同部分被连接起来，以执行所需的操作。 
			 例如，如果要求完成一个加法运算，[算术逻辑单元](https://zhida.zhihu.com/search?content_id=185093927&content_type=Article&match_order=1&q=%E7%AE%97%E6%9C%AF%E9%80%BB%E8%BE%91%E5%8D%95%E5%85%83&zhida_source=entity) ALU 将被连接到一组输入和一组输出，输入端提供需要相加的数值，输出端将含有最后的运算结果。
		根据指令需要，有可能要访问主存，读取操作数，这样就进入了访存取数（Memory，MEM）阶段。
			根据指令地址码，得到操作数在主存中的地址，并从主存中读取该操作数用于运算。
		结果写回（Writeback，WB）
			把执行指令阶段的运行结果数据“写回”到某种存储形式
			结果数据经常被写到CPU内部寄存器中，以便被后续的指令快速地存取；
			但在有些情况下， 结果数据也可被写入相对较慢、但较廉价且容量较大的主存。
		在指令执行完毕、结果数据写回之后，若无意外事件（如结果溢出等）发生，计算机就接着从程序计数器PC中取得下一条指令地址，开始新一轮的循环，下一个指令周期将顺序取出下一条指令。
	指令流水线技术
		==**在任一条指令的执行过程中，各个功能部件都会随着指令执行的进程而呈现出时忙时闲的现象。**==
		为了加快计算机的工作速度，就应使各个功能部件并行工作，即以各自可能的高速度同时、不停地工作，==**使得各部件的操作在时间上重叠进行，实现流水式作业**==。
		于是指令流水线技术应运而生
			从原理上说，计算机的流水线（Pipeline）工作方式就是将**一个计算任务细分成若干个子任务**，**每个子任务都由专门的功能部件进行处理**，一个计算任务的各个子任务由流水线上各个功能部件轮流进行处理
			这样，**不必等到上一个计算任务完成， 就可以开始下一个计算任务的执行**。
			![[Pasted image 20260214161903.png]]
	三大总线
		系统总线、片内总线、通信总线
			片内总线的含义最稳定。片内总线指的是同一颗芯片内部的互连，它把==**取指译码、乱序窗口、执行单元、L1 寄存器和 L2 寄存器这些模块连起来**==
			系统总线更像“把处理器这一侧和整机资源连起来”的那层互连。例如连接DDR内存
				一类是连接内存，另一类是通向芯片组（主板）或其他处理器的高速互连。
			通信总线更偏向“CPU 和外部设备或外部系统之间的数据交换通道”，它强调跨模块、跨芯片甚至跨板级的兼容和连接距离，所以它更常对应标准化的 I O 互连，比如 PCIe、USB、SATA、以太网，也包括多路 CPU 之间的互连链路
	在指令流水线技术的影响下，对存储的容量要求进一步提升
	为了容纳更多存储，虚拟存储器应运而生
		1950 到 1960 年代主存非常贵，程序规模却快速变大。没有虚拟存储器时，开发者常用覆盖装入：程序自己决定哪段代码或数据先放内存，哪段暂时放外存，需要时再手工搬运。这会把大量精力耗在手工规划与反复重编译上。
		虚拟存储器的出现，本质是在系统层面把这件事自动化：把地址空间切页，只有被访问的页才调入内存，让外存成为后备仓库，从而用更小的主存跑更大的程序，并减少人工管理的负担。
		详细地说：
			计算机的操作系统通过软硬件结合的方式，将程序使用的内存地址 （称为虚拟地址 ）映射到计算机内存中的物理地址 。
			![[Pasted image 20260214165115.png]]
			从进程或任务的角度看， ==**主存储器呈现为一个连续的地址空间或多个连续段的集合。操作系统负责管理虚拟地址空间 ，并将实际内存分配给虚拟内存。**==
			CPU 中的地址转换硬件（通常称为内存管理单元 ，MMU）会自动将虚拟地址转换为物理地址。操作系统内部的软件可以扩展这些功能，例如利用磁盘存储来提供超过实际内存容量的虚拟地址空间，从而引用比计算机物理内存更多的内存。
			而且==虚拟存储器的存储容量取决于计算机的寻址能力==
			==虚拟存储器有相当于外存的容量，也有接近于主存的访问速度==
	在流水线工作方式下，==**进程调度是操作系统做的“分配 CPU 时间”的决策**==。操作系统面对很多个进程或线程，必须决定下一小段时间把 CPU 交给谁运行，以及什么时候把正在运行的那个换下来。
	

每次循环结尾都会有一个分支：要么跳回循环头继续执行，要么退出循环。流水线越深，越需要提前知道这个分支是“继续”还是“退出”，否则每一轮循环都可能在分支点停一次，吞吐就很惨。所以现代 CPU 会对“后向跳转”这类循环分支做高准确率预测，尽量让前端不断供。

存储数据的方式
	在二进制的基础上，我们想要表示正负数，但是0,1怎么表示正负呢？
	于是我们加一个比特位用于表示正和负这两种状态
	那么问题就来了，这样0就有两种表现方式了：
	0000 0000和 1000 0000也就是+0和-0

	==反码==：为了解决原码不能计算负数而出现的。  
	==计算规则==  
		**正数**：因为反码是为负数所设计的所以正数的**反码等于原码**。  
		**负数**：符号位不变（为了跟正数区分），其它位取反，0变1，1变0。例如：-1的反码为1111 1110。
	但是这带来了新的问题
		-3+5是要等于2的，但是用补码计算结果为1，因为0计算了两次（如图所示），
		所以我们应该把所有的负数加一，就出现了补码，此时-0为（1）0000 0000，+0为0000 0000，就很好地解决了0可能计算两次的问题。

	在计算机中通常用二进制的补码存储整数、用移码进行浮点运算
	在计算机内部只用二进制表示指令和数据

操作系统革命
	最早的时候没有真正意义上的操作系统。人把程序和数据手工装进去，按开关、打孔卡、接线，机器跑完再换下一份。
	问题很明显：人和机器都在等，尤其是机器在等人的时间太多，计算资源浪费得离谱。
	![[Pasted image 20260215144946.png]]
	后来出现==**单道批处理**==。思路是把==**一堆作业排队，一次交给机器，让机器按顺序一个个跑完**==。这样“人不用一直守着机器”，效率比纯手工高，但仍然有一个大问题：程序运行过程中经常要等 I/O，比如读卡、读磁带、打印输出。==**CPU 等 I/O 的时候还是在发呆，因为内存里只有这一道作业，没别的活可切换。**==
	多道批处理。核心改进只有一句话：==**内存里同时放多道作业，一个作业去等 I/O 时，系统就把 CPU 切给另一个作业继续算。**==这样 CPU 的空闲时间大幅减少，整机吞吐就上来了。你可以把它理解成“厨房里同时做多桌菜，这桌在炖汤就先炒另一桌的菜”，总之别让灶台停着。
		==**多道程序设计技术指的是一种通用技术，也就是让多个程序同时驻留在内存里，操作系统在它们之间进行切换。（即多个进程并发执行）**==这样一来，当某个程序因为输入输出而阻塞，操作系统就可以把 CPU 让给另一个就绪程序，从而提高 CPU 利用率和系统吞吐量。这个概念强调的是“怎么做”，也就是调度、内存管理、保护与中断这些机制配合起来，让多道并发成为可能。
		注意，多道程序设计技术和多道批处理技术不是同一个概念
	但是==**批处理还有一个根本缺陷：它不适合交互**==。用户想敲一条命令马上看到反馈，批处理的排队方式让人等得很难受。
	于是出现分时操作系统。==**它把 CPU 时间切成很小的时间片，轮流分给多个用户或多个程序，让每个人都感觉自己在“独占”机器。**==分时解决的是“响应速度”和“交互体验”，让计算机从“后台算账机器”变成“可以对话的工具”。
	与此同时，还有一类场景比交互更苛刻：不是要“快一点反馈”，==**而是要“必须在规定时间内完成”**==，例如==**工业控制、飞行控制、汽车控制**==。于是出现实时操作系统。它关心的不是平均速度，而是截止时间能不能保证，最坏情况下会不会失控。为了保证确定性，它会强调优先级、抢占、可预测的调度与资源管理，==**宁可牺牲一些总体吞吐，也要把关键任务的时间约束守住。**==
		实时系统分为：过程控制系统、信息查询系统、事务处理系统
			工业生产自动控制系统、航天器飞行控制和发射控制系统都属于过程控制系统
			仓库管理系统、图书资料查询系统属于信息查询系统
			订票系统、银行管理系统属于事务处理系统
	操作系统最怕两类并发干扰：一类来自多个进程或线程同时访问同一份共享数据，另一类来自中断在任意时刻打断当前执行。
		==**原语的功能就是把一些关键更新做成原子动作**==，也就是一串步骤对外看起来像“一步完成”。很多课程会把这种性质概括成“原语执行期间不允许被打断”，本质是保证原子性与一致性。原子操作在操作系统课里通常就被定义为“不能被中断的一系列操作”。
		你可以把原语理解成两大类最常见的内核底层积木。  
			第一类是进程控制原语，典型就是阻塞和唤醒以及切换进程状态。
			比如把一个进程从就绪队列移走放进等待队列，这个动作如果被打断或被并发修改，就会把队列结构弄乱。Linux 的讲义里甚至直接把 wake_up 叫作 primitive，并说明它会把等待队列里的任务转到可运行队列。
			第二类是同步互斥原语，典型就是对信号量或锁的 wait 与 signal 这一类操作。它们保证多个执行流对临界资源的进入与退出不会互相踩踏。
		例如：一个进程 A 在读磁盘文件。磁盘读很慢，A 不能一直占着 CPU 等，于是 A 要把自己“睡眠”，等磁盘中断来了再被唤醒。
			系统里有两个共享结构：就绪队列 ReadyQ 和等待队列 WaitQ。
				正常逻辑是：
				1）A 发起 I/O 请求，发现数据没好。  
				2）操作系统执行阻塞原语，把 A 从 ReadyQ 摘掉，插入 WaitQ，并把 A 的状态改成 Waiting。  
				3）调度器挑另一个进程 B 运行。  
				4）磁盘 I/O 完成产生中断。中断处理里执行唤醒原语，把 A 从 WaitQ 摘掉，插回 ReadyQ，并把 A 状态改成 Ready。
				关键点是第 2 步和第 4 步都必须不可分割，否则会出现“丢失唤醒”这种经典错误。
				我用一个具体的“被打断”的时间线演给你看：
					- 时刻 t0：A 正在执行“阻塞”的那段代码，刚刚把自己的状态改成 Waiting，但还没来得及把自己挂进 WaitQ。
					- 时刻 t1：磁盘刚好在这时完成，产生中断。中断处理程序跑起来，执行“唤醒 A”。它去 WaitQ 里找 A，结果找不到，因为 A 还没挂进去。于是中断处理认为“没有人等待”，就结束了。
					- 时刻 t2：中断结束，A 继续执行阻塞流程，把自己挂进 WaitQ，然后调用调度切换出去。
					- 结果：A 现在永远睡在 WaitQ 里，因为那个“唯一一次”的 I/O 完成通知已经错过了。系统就卡死或出现莫名其妙的长期阻塞。
				这个 bug 的根本原因是：==**阻塞过程“改状态 + 入队”不是原子动作，被中断插进来导致唤醒时看到的是不一致的中间状态。**==
		原语的类型
			进程控制类原语主要在“进程状态变化”和“调度切换”时用  
			1）创建原语，常见语义是创建一个新进程或新线程并加入就绪队列。比如你启动一个新程序，或者浏览器新开一个渲染进程时。  
			2）撤销原语，常见语义是==**结束进程或线程**==，回收它占用的资源并从系统结构里移除。它会在程序正常退出或异常崩溃时触发。  
			3）阻塞原语，常见语义是把==**当前运行实体从运行态转为等待态**==，并从就绪队列移到某个等待队列。它会在“必须等某个事件才能继续”的时候触发，比如读文件时数据还没到、等待网络包、等待锁、等待某个条件变量。  
			4）唤醒原语，常见语义是把等待队列中的一个或多个实体移回就绪队列。它会在事件发生时触发，比如磁盘中断表示 I/O 完成、网卡中断表示数据到达、释放锁让等待者可继续。  
			5）调度与切换原语，常见语义是选择下一个要运行的线程并完成上下文切换，也就是保存当前寄存器现场并恢复下一个线程现场。它会在时间片用完、当前线程阻塞、出现更高优先级线程就绪等时机触发。
	创建到就绪
		内核创建进程控制块并初始化资源，然后把==**进程插入就绪队列**==。
		这个阶段用在启动程序、创建子进程、服务端接到连接派生处理进程等场景。
	就绪到运行
		调度器从==**就绪队列挑一个进程**==，把 CPU 交给它，发生一次上下文切换。这个阶段每次调度都会用到。
	运行到就绪
		两类常见触发：
		==**时间片用完被抢占，进程被放回就绪队列**==
		==**进程主动让出 CPU，例如调用让出处理器的接口**==
		这个阶段用在分时系统里保证公平性，也用在高优先级任务到来时的抢占。
	运行到阻塞
		进程在运行时遇到必须等待的事件，==**内核把它从运行位置移到等待队列。**==典型触发：同步读文件等待磁盘、等待锁、等待信号量、等待某个条件变量、等待子进程结束、睡眠到某个时间点。
		这个阶段用在“进程继续跑也没意义”的时候，因为它缺少继续执行所需的条件。
	阻塞到就绪
		等待的事件发生以后，中断处理或内核唤醒逻辑把进程从等待队列搬回就绪队列。
		典型触发：IO 完成中断、网络中断、定时器中断、释放锁或发信号量的那一方执行唤醒。
		这个阶段用在“条件已满足，进程重新具备运行资格”的时候。
	

操作系统的功能和任务有：进程管理（又称处理机管理）、存储管理、设备管理、文件管理和用户接口
	程序本身只是磁盘上的一份静态文件，==**进程则是这份程序被操作系统加载进内存以后，带着它自己的运行现场一起在 CPU 上执行的“活体”。**==
		假设你一边用浏览器看网页，一边用播放器放音乐，同时你还开着一个下载工具在下载文件。
		*浏览器是一个程序，但当你打开浏览器后，操作系统会创建浏览器进程。*
		*浏览器进程里又可能有多个子进程或线程，比如页面渲染、网络请求、插件隔离等，这些是实现细节，你现在先把它整体看成一个进程也没问题。*
		*播放器同样是一个进程。它会反复做两件事：从磁盘或网络读取音频数据，然后把音频送到声卡缓冲区。读取数据时它经常会等待 IO，于是它会进入阻塞状态。等数据到了，它就回到就绪状态等待 CPU，再继续处理。*

直接寻址是“指令给出操作数地址”。也就是说，指令的地址字段 A 就是操作数所在的有效地址 EA，所以处理器取数时会直接去内存地址 A 取。用公式写就是
间接寻址是“指令给出的是一个地址的地址”。也就是说，指令的地址字段 A 指向一个内存单元，这个内存单元里存放的内容才是真正的有效地址 EA。

取指令的地址重定位
	地址重定位的本质是：系统把程序使用的地址变成内存里真正要访问的位置，也就是==**把逻辑地址或虚拟地址映射到物理地址**==。只不过“虚拟地址”是现代系统更常用的说法，“逻辑地址”是更早期或更泛化的说法。
	静态地址重定位和动态地址重定位虽然都是地址重定位但是方式不同
		我们假设：$$物理地址 = base(基本地址)+逻辑地址$$
		某个进程在内存里的起始位置。更具体一点，base 是地址转换时用的那个起点，它告诉系统：这个进程的逻辑地址 0 对应到物理内存的哪个位置。
		静态地址重定位
			装入器把程序放到某个起始位置 base 以后，把程序里所有需要修补的地址都做一次替换，让它们变成对应的物理地址。
			等修补完成，CPU 执行时就不需要再做地址转换了。它的核心特征是一次性完成，之后地址被写死，所以它天然只能在装入阶段做。
		动态地址重定位
			==程序里仍然保留逻辑地址或虚拟地址==，CPU 每次取指令或读写数据都要经过一次转换，==转换规则由硬件寄存器或页表提供。==它的核心特征是持续发生，属于运行时机制，所以必然是在程序执行期间进行。
			从而动态地址重定位==**不需要程序装入连续固定的内存空间**==

CPU的算力分配
	==**成操作系统给某个进程的一段“连续使用 CPU 的最长时长”。**==
	这段时长用完以后，==操作系统就会把 CPU 收回来==，让别的进程也有机会运行。
	它的目的不是让进程更快完成，而是让多道程序看起来“同时在跑”，并且让交互更流畅。
	时间片大规模出现的背景是分时操作系统。
		早期的单道批处理更像是一个作业占着机器一直跑完，后来为了让一台机器能服务多个用户和多个任务，系统就需要一种公平的轮流机制。于是抢占式调度配合时间片就变成了很核心的做法：没有人能一直霸占 CPU，哪怕它一直不阻塞也不主动让出。
	时间片具体做什么  
		==**操作系统会给正在运行的进程设置一个倒计时**==。
		硬件定时器周期性产生中断，内核在中断里检查该进程是否用完了自己的时间片。
		用完了就触发一次调度：==**把该进程从运行态切回就绪队列，再从就绪队列选另一个进程上 CPU**==。你前面那道题里“运行态时间片用完转就绪态”说的就是这一步。

路径名属于文件系统命名与查找这条链路里的输入，PCB 表属于进程管理这条链路里的账本。
在操作系统里，把“文件名”一步步转成“文件在磁盘上的存储地址”，靠的就是文件目录结构。
==**进程控制块PCB是进程存在的唯一标志**==

进程是指一个具有一定独立功能的程序关于某个数据集合的一次运行活动。
简单地说，==**进程是可以并发执行的程序的执行过程**==，它是控制程序管理下的基本的多道程序单位。
同一个程序，可以同时对应多个进程。
例如你同时开两个微信，==**这两个进程用的是同一个程序文件，但它们各自的运行状态、内存数据、打开的资源都不同。**==
这就说明：
“程序” ≠ “进程”
“内存中的程序”也 ≠ “进程”

进程具有动态性、共享性、独立性、制约性、并发性
	第一阶段是创建，也就是进程出生。
		你双击浏览器图标的那一刻，操作系统会做三件关键事。
		①操作系统会为它分配一个进程标识。
		②操作系统会建立一份进程控制块 PCB，用来记录它的状态、优先级、打开的资源、内存映射等。
		③操作系统会把程序的代码和必要的数据映射进内存，把它放进就绪队列等待被调度。
		==这一步体现了进程的动态性==
	第二阶段是就绪到运行，也就是进程第一次上 CPU。
		调度器从就绪队列挑一个进程上 CPU，比如浏览器先上。
		浏览器开始执行时会不断触发两种事情。
			第一种事情是纯计算，比如排版、脚本执行。
			第二种事情是发起 IO，比如请求网页、读取缓存。
		==这个阶段最能体现并发性，因为音乐播放器和下载工具并没有消失，它们也在就绪队列里等 CPU。==
	第三阶段是运行到阻塞，也就是进程主动“去等”。
		浏览器向网络请求数据以后，浏览器进程此时继续占着 CPU 也没有意义，因为数据还没有回来。
		操作系统会把浏览器从运行态移到阻塞态，把它挂到某个等待队列里，例如等待网络包、等待磁盘读完、等待定时器到点。
		然后 CPU 会立刻让给下载工具或播放器。
		==这一步体现了制约性，因为进程的推进受制于外部事件和资源是否就绪，进程必须等待条件满足才能继续。==
	第四阶段是阻塞到就绪，再到运行，也就是被唤醒并继续执行。
		等网卡中断告诉系统数据到了，操作系统把浏览器从等待队列移回就绪队列。
		浏览器什么时候再次变成运行态取决于调度器。
		你会看到同一个进程在整个生命周期里不断经历就绪、运行、阻塞这三态的循环。
		这一步把动态性展示得非常直观，因为进程的状态是持续变化的，而不是一成不变的。
	第五阶段是退出，也就是进程死亡。
		你关闭下载工具窗口时，下载进程会走到退出路径。
		操作系统会回收它占用的内存，关闭它打开的文件和网络连接，释放它申请的内核对象，然后删除 PCB。
		你会发现退出时系统会非常强调“资源回收”，这就是进程作为资源管理单位的原因。
	==独立性对应的是每个进程拥有相对独立的运行环境。操作系统通常给每个进程独立的地址空间和资源记录，所以浏览器崩了不一定把音乐播放器带崩。==
	==共享性对应的是进程之间可以共享某些资源，但共享必须受操作系统管理。==多个进程可以共享同一个磁盘文件的内容，可以共享同一个打印机队列，可以共享只读代码段，还可以通过共享内存实现高速通信。
	制约性对应的是进程推进必须遵守调度和同步规则。进程会受 CPU 时间片约束，受优先级约束，受锁和信号量约束，受 IO 完成约束，受内存和句柄数量约束。
	==**处于运行状态的程序在时间片用完之后转为就绪状态**==
	==**进程状态PCB是进程存在的唯一标志**==

连续存储管理技术指的是，**==操作系统在把一个进程装入内存时，要求这个进程的整个地址空间在物理内存里占用一段连续的区域。==** 。也就是说，一个进程进入内存后，它的代码段、数据段、堆栈等在物理地址上不会被拆成好几块散落开来，而是连成一整段。
	第一种是单一连续分配，也常被说成单分区或单用户连续分配。内存里通常固定划出一部分给操作系统，其余部分一次只装入一个用户程序。
	第二种是可变分区式分配，也就是多分区连续分配。内存中同时驻留多个进程，每个进程各占一段连续区域。

文件
	文件是一组带标识、具有完整逻辑意义的相关信息的集合。
	有三个关键要素：**==它必须能被区分、它承载信息、它在语义上是一个“整体”==**
	文件属性包括文件类型、文件名称、文件长度、文件的物理地址、文件的建立时间等
	属性是描述文件的，内容是文件承载的数据本身。


I/O 的程序查询方式指的是，**==处理器用一段循环程序不断读取设备控制器的状态寄存器，反复判断“设备是否就绪”==**。一旦就绪，处理器就通过读写数据寄存器把数据搬运到内存或从内存送到设备。
